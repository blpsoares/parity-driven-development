# Contributing to PDD

Thanks for your interest in improving **PDD â€” Parity-Driven Development**. This guide covers how to propose changes, how the repo is laid out, and how to run the tests and build.

Please also read [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) before participating, and [DEVELOPMENT.md](DEVELOPMENT.md) for a deeper walkthrough of the local development workflow.

## Proposing changes

1. **Fork** the repository and clone your fork.
2. Create a **branch** off `main` with a descriptive name (e.g. `fix/status-panel-width`, `feat/adapt-windsurf`).
3. Make your change, keeping commits focused and self-contained.
4. Run the tests and the build locally (see below) before opening a pull request.
5. Open a **pull request** against `main`, describing what changed and why.

### Commit messages

Commits follow [Conventional Commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `build:`, and so on. Example:

```
feat(adapt): add windsurf harness target
fix(tui): clamp panel width on narrow terminals
docs: clarify qa-local gate in audit-pr
```

## Repository layout

| Path | What it is |
| --- | --- |
| `skills/` | The PDD commands, one directory per command, each holding a `SKILL.md`. These are the `audit-*` commands (`audit-bootstrap`, `audit-new`, `audit-investigate`, `audit-resolve`, `audit-compare`, `audit-qa`, `audit-pr`, `audit-status`). |
| `scripts/pdd/` | TypeScript source for the `pdd` dashboard CLI (`index.ts` plus modules like `state.ts`, `tui.ts`, `render.ts`, `adapt.ts`, `i18n.ts`, `prompt.ts`, `update.ts`, with co-located `*.test.ts` files). |
| `dist/pdd.js` | The built, bundled CLI â€” a single ESM file with a Node shebang. Generated; do not hand-edit. |
| `docs/` | Single-source markdown documentation (`concepts/`, `guides/`, `reference/`, `superpowers/`) consumed by the site. |
| `demo/` | The VHS demo â€” `pdd.tape` (the [VHS](https://github.com/charmbracelet/vhs) script), `seed.sh`, and the generated `pdd.gif`. |
| `bin/`, `hooks/`, `scripts/*.sh` | CLI entry point, harness hooks, and install/update shell helpers. |
| `AGENTS.md` | The portable, harness-agnostic agent guide describing the method and the commands. |

## Running tests

Tests live next to the source in `scripts/pdd/` and run with Bun:

```bash
bun test scripts/pdd
```

This is the same as `bun run test` (see `package.json`).

## Building

The CLI is bundled with [esbuild](https://esbuild.github.io/) into `dist/pdd.js`:

```bash
bun run build      # or: npm run build
```

Under the hood this runs:

```
esbuild scripts/pdd/index.ts --bundle --platform=node --format=esm --target=node18 \
  --outfile=dist/pdd.js --banner:js="#!/usr/bin/env node"
```

The output `dist/pdd.js` runs on **Node (>= 18) or Bun** â€” no `npm install` is required to run it. When you change anything under `scripts/pdd/`, rebuild so `dist/pdd.js` stays in sync, and include the rebuilt bundle in your commit.

You can also run the CLI directly from source without building:

```bash
bun run pdd        # runs scripts/pdd/index.ts
```

## Contribution guidelines

### Skills are portable English markdown

Each command is a `skills/<name>/SKILL.md` file. The **content is portable markdown written in English** â€” it must be self-contained instructions that any coding agent can follow, independent of harness. Only the *registration* format differs per harness, and that is generated by `pdd adapt <harness>` (which writes into each agent's native skills directory). When editing a skill:

- Keep the body harness-agnostic; do not hardcode anything specific to a single agent.
- Use `$ARGUMENTS` where a command takes arguments.
- Preserve the inviolable rules: the AI never authors commits; `push`/`gh pr create` happen only after an explicit human "yes"; merge is 100% human, only after target-environment QA approves.

### Docs are single-source markdown

The files under `docs/` are the single source of truth for documentation and are consumed directly by the site. Write plain markdown, keep one topic per file, and update the docs in the same PR as any behavior change they describe.

### Code style

- Write **TypeScript** with strict typing.
- Follow the existing lint/formatting conventions in the repo.
- Add or update tests for any behavior change, and keep `bun test scripts/pdd` green.
